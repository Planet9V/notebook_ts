# Database - Hybrid Persistence Configuration

Open Notebook employs a **hybrid polyglot persistence architecture** that leverages the specific strengths of two distinct database technologies to optimize transactional document graphing and high-speed semantic search caching:

1. **SurrealDB (v2):** Used for relational, document, and graph-like transactional data. It maintains user accounts, Kanban pipeline cards, notes, activity feeds, and system logs.
2. **PostgreSQL (pg17 with pgvector):** Used for high-speed semantic search caching (`ResearchMemory`) for external Valyu/Brave searches.

## 1. SurrealDB Configuration (Core Transactional)

SurrealDB acts as the core database of record. It handles all workspace transactions, graph relationships (e.g. linking notes to customers via `entity_note` edge tables), and stores primary document and note embeddings.

### Connection Environment Variables
```env
SURREAL_URL="ws://surrealdb:8000/rpc"
SURREAL_USER="root"
SURREAL_PASSWORD="root" # Change in production!
SURREAL_NAMESPACE="open_notebook"
SURREAL_DATABASE="open_notebook"
```

### High-Dimensional Vector Support
* **Capacity:** Stores high-dimensional vector representations natively as `array<float>`.
* **No Dimension Limit:** SurrealDB does not impose strict page size limits on vector columns. This allows it to seamlessly store and search the **2,560-dimension vectors** generated by the default local embedding model (`qwen3-embedding:4b`).

## 2. PostgreSQL + pgvector Configuration (Semantic Cache)

PostgreSQL (with `pgvector` extension) acts as a high-speed semantic search caching layer. It caches external search results (from Brave Search and Valyu APIs) to prevent slow external API lookups and isolate cache writes from SurrealDB. It can store tables, pgvector embeddings, and file registries.

### Connection Environment Variable
```env
POSTGRES_DSN="postgresql://tetrel:tetrel_dev@postgres:5432/tetrel_research"
```

### HNSW Dimension Limits (The 8KB Problem)
Standard PostgreSQL installations are compiled with an **8KB (8,192 bytes) page size limit**. When building Hierarchical Navigable Small World (HNSW) indexes, the vector data and its index metadata must fit inside a single page:
* **Standard `vector` (32-bit float):** Hard-limited to **2,000 dimensions** ($2,000 \times 4\text{ bytes} = 8,000\text{ bytes}$).
* **`halfvec` (16-bit float):** Hard-limited to **4,000 dimensions** ($4,000 \times 2\text{ bytes} = 8,000\text{ bytes}$).
* **Impact on 4,096D embeddings:** 4,096-dimensional vectors cannot have HNSW indexes created on them in standard environments (unless compiled with custom 16KB/32KB block sizes).
* **The Cache Solution:** For PostgreSQL, Open Notebook uses standard **1,536-dimension vectors** (compatible with OpenAI `text-embedding-3-small` or Matryoshka dimension-reduced vectors), which fit comfortably within the 2,000 HNSW limit.

## 3. Agent Guidelines: Index Evaluation & Search Optimization

Autonomous agents working on this codebase should actively evaluate and optimize PostgreSQL search queries and pgvector indexes based on the following rules:

### A. Indexing Strategy Check
Every table storing vectors in PostgreSQL must have a matching HNSW index to guarantee sub-millisecond search performance. For example, the `research_corpus` table uses:
```sql
CREATE INDEX IF NOT EXISTS idx_research_embedding 
    ON research_corpus USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```
Ensure that:
* Vector columns never exceed `vector(2000)` if an HNSW index is to be constructed.
* If a model outputs >2000 dimensions, either apply dimensionality reduction (e.g. Matryoshka) before insertion or avoid using HNSW (flat scan) if the table size is small.

### B. Hybrid Search Optimization via Reciprocal Rank Fusion (RRF)
Do not rely solely on vector search or keyword search alone. For high-quality RAG and search caching, write queries that merge semantic search (using pgvector `<=>` cosine distance) and keyword search (using a GIN index on `to_tsvector`) via the RRF formula:

```sql
WITH semantic_search AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $1::vector) as rank
    FROM research_corpus
    WHERE embedding IS NOT NULL AND 1 - (embedding <=> $1::vector) > $2
    LIMIT $3
),
keyword_search AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(to_tsvector('english', content), plainto_tsquery('english', $4)) DESC) as rank
    FROM research_corpus
    WHERE to_tsvector('english', content) @@ plainto_tsquery('english', $4)
    LIMIT $3
)
SELECT rc.*, 
       COALESCE(1.0 / (60 + s.rank), 0.0) + COALESCE(1.0 / (60 + k.rank), 0.0) AS rrf_score
FROM research_corpus rc
LEFT JOIN semantic_search s ON rc.id = s.id
LEFT JOIN keyword_search k ON rc.id = k.id
WHERE s.id IS NOT NULL OR k.id IS NOT NULL
ORDER BY rrf_score DESC
LIMIT $3;
```
*Agents must ensure new schemas and search methods adhere to this hybrid retrieval model to maximize relevance.*
